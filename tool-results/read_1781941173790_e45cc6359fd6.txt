     1→# Универсальный скрапер медицинских прайсов — методология
     2→
     3→> **Главная цель:** построить универсальный движок, способный автоматически
     4→> собирать данные о ценах и услугах с **любого** медицинского сайта — независимо
     5→> от архитектуры, фреймворка, способа хранения данных и наличия антибот-защиты.
     6→> Движок **сам** определяет тип сайта, выбирает оптимальный метод сбора,
     7→> обнаруживает страницы с ценами, выводит унифицированную структуру и
     8→> самовосстанавливается при изменении вёрстки.
     9→>
    10→> Документ уровня senior fullstack + devops. Описывает автоопределение тиров,
    11→> стратегии discovery, адаптивный парсинг, AI-ассистентов (VLM/LLM) и операционные
    12→> практики. Разведанные сайты (Veramed, Gemotest, Helix, Altamed+, Medsi, CMD)
    13→> используются как **эталонные кейсы** для валидации универсальности движка.
    14→
    15→---
    16→
    17→## 0. Scope сбора (бизнес-ограничение — НЕ изменять без согласования)
    18→
    19→### 0.1. Принцип
    20→
    21→Система отслеживает цены **только в одном выбранном географическом регионе**:
    22→либо конкретный город (например, Москва), либо Московская область целиком.
    23→**Полная федеральная тарифная сетка конкурента не собирается.**
    24→
    25→### 0.2. Обоснование
    26→
    27→1. **Бизнес-смысл:** конкурентное ценообразование локально. Цены Gemotest в
    28→   Новосибирске не релевантны для клиники в Одинцово.
    29→2. **Этическое:** меньшая нагрузка на сайты конкурентов. Сбор 1 города вместо 17
    30→   снижает трафик в 17 раз.
    31→3. **Юридическое:** минимизация объёма собираемых данных снижает риски претензий
    32→   по ToS.
    33→4. **Cost:** VLM/Playwright нужны только в fallback; минимальный сбор = минимальный
    34→   бюджет на прокси и AI Skills.
    35→
    36→### 0.3. Target regions (по умолчанию)
    37→
    38→| Регион | Что входит | Заметка |
    39→|---|---|---|
    40→| **Москва** (город) | ЦАО, САО, ВАО, ЮАО, СВАО, ЮВАО, ЗАО, СЗАО, ЮЗАО, ТАиО, ЗелАО | Основной регион для большинства федеральных сетей |
    41→| **Московская область** | Города МО: Одинцово, Долгопрудный, Балашиха, Химки, Мытищи, Красногорск, Реутов, Домодедово, и т.д. | Для региональных клиник (Veramed, Altamed+) |
    42→
    43→**Конфигурация region** задаётся через переменную окружения `TARGET_REGION` и
    44→хранится в БД (`AppConfig.targetRegion`). Смена региона = миграция данных
    45→(старые snapshot'ы остаются, новые собираются с другим scope).
    46→
    47→### 0.4. Влияние на архитектуру
    48→
    49→#### 0.4.1. Унифицированный фильтр на уровне Fetcher
    50→
    51→Каждый Fetcher получает `region: Region` параметр. На уровне spec-схемы для
    52→каждого конкурента определён `region_strategy` — как применить регион к URL:
    53→
    54→```yaml
    55→region_strategy:
    56→  type: url_path_segment     # auto-detected (см. раздел 2)
    57→  param: city_slug
    58→  mapping:
    59→    moscow: msk
    60→    spb: sankt-peterburg
    61→    mo: ""
    62→```
    63→
    64→Поддерживаемые типы region_strategy (выявлены из разведанных сайтов):
    65→
    66→| Тип | Пример | Где применяется |
    67→|---|---|---|
    68→| `url_prefix` | `/{city}/catalog/...` | Gemotest |
    69→| `url_path_segment` | `/catalog/{city}/{slug}/` | CMD |
    70→| `url_subdomain` | `https://{city}.medsi.ru/...` | Medsi |
    71→| `url_query` | `/services/x/?clinic={slug}` | Medsi (клиники Москвы) — **запрещено robots.txt** |
    72→| `cookie` | `Cookie: cityId={id}` | Helix |
    73→| `ip_default` | SSR определяет по IP | Helix (нельзя изменить без прокси) |
    74→| `tariff_select` | Вкладка тарифа в HTML | Veramed (3 тарифа клиник МО) |
    75→| `none` | Регион не влияет | Altamed+ (одна локация — МО) |
    76→
    77→**Все типы auto-detectable** через Probe Engine (см. раздел 2).
    78→
    79→#### 0.4.2. Хранение region в данных
    80→
    81→```prisma
    82→model PriceSnapshot {
    83→  // ...
    84→  region       String   @default("moscow")  // moscow | mo | spb | ...
    85→  locationKey  String?  // gemotest: "moskva"; medsi: "klinika-na-leninskom"; veramed: "premium"
    86→  // ...
    87→  @@index([serviceId, region, scrapedAt])
    88→}
    89→```
    90→
    91→`locationKey` — детализация внутри региона (конкретная клиника/тариф/категория).
    92→Для большинства конкурентов = `null` (одна цена на регион). Для Medsi — slug
    93→клиники в Москве. Для Veramed — название тарифа.
    94→
    95→#### 0.4.3. Пересмотр нагрузки (ТОЛЬКО целевой регион)
    96→
    97→| Сайт | Без scope (вся РФ) | Со scope (1 регион) | Снижение |
    98→|---|---|---|---|
    99→| Veramed | 1 URL × 3 тарифа = 3 261 услуг | 1 URL × 3 тарифа МО = **3 261 услуга** (Veramed — уже только МО) | 0% (уже локальный) |
   100→| Gemotest | 66 URL × ~17 городов = ~56 000 услуг | 66 URL × 1 город = **~3 300 услуг** | **-94%** |
   101→| Helix | 90 URL × SSR по IP (только СПб из РФ-сервера) | 90 URL = **1 078 услуг** для одного города | 0% (нужен прокси для других городов) |
   102→| Altamed+ | 294 страницы | 294 страницы (только МО) = **~900 услуг** | 0% (уже локальный) |
   103→| Medsi services | 1 248 × 63 клиники = ~78 000 страниц | 1 248 × **1 город (SEO-блок)** = **~1 250 услуг** | **-98%** |
   104→| Medsi labmarket | 1 086 × 17 городов | 1 086 × 1 город = **1 086 анализов** | **-94%** |
   105→| CMD | 132 866 URL (86 городов) | 1 510 URL (msk) = **1 510 анализов** | **-99%** (цены одинаковы по городам) |
   106→
   107→**Итог: с scope-фильтром полный сбор = ~11 385 позиций вместо ~80 000+.**
   108→Объём трафика: ~1.1 ГБ вместо ~5+ ГБ.
   109→
   110→#### 0.4.4. Политика для много-клиниковых сетей (Medsi)
   111→
   112→Для Medsi в пределах Москвы доступно 63 клиники, но:
   113→- `?clinic=` запрещён в robots.txt
   114→- SEO-блок даёт только min-цену по всем клиникам Москвы
   115→
   116→**Решение:** собираем min-цены из SEO-блока. Если для конкретной услуги
   117→критична цена по конкретной клинике → ручной lookup через UI (не в scope
   118→автоматического сбора).
   119→
   120→#### 0.4.5. Мультигородность на будущее
   121→
   122→Если бизнес-требования изменятся и понадобится несколько регионов:
   123→1. Добавить `targetRegions: []` в AppConfig (вместо одного `targetRegion`)
   124→2. Scheduler запускает N параллельных job'ов — по одному на регион
   125→3. Каждый job использует свой `region_strategy` mapping
   126→4. **Не удалять** старые snapshot'ы — они остаются как исторические данные
   127→
   128→Это расширение заложено в архитектуру, но **не реализуется в MVP**.
   129→
   130→### 0.5. Антипаттерны (запрещено)
   131→
   132→1. ❌ Собирать цены по всем городам «на всякий случай» — нарушение scope
   133→2. ❌ Использовать `?clinic=` для Medsi даже если очень хочется — нарушение robots.txt
   134→3. ❌ Менять `TARGET_REGION` без миграции БД (старые snapshot'ы привязаны к старому region)
   135→4. ❌ Хранить цены без поля `region` — невозможно сопоставить данные
   136→5. ❌ Смешивать в одном scrape-run два региона — данные нельзя сравнивать между собой
   137→6. ❌ Писать парсер под конкретный сайт без auto-detection — движок должен быть universal
   138→
   139→### 0.6. Валидация scope в pipeline
   140→
   141→```typescript
   142→// В FetcherOptions — обязательное поле
   143→interface FetcherOptions {
   144→  region: Region;            // 'moscow' | 'mo' | 'spb' | ...
   145→  locationKey?: string;      // детализация внутри региона
   146→  // ...
   147→}
   148→
   149→// В Validator — проверка консистентности
   150→function validateSnapshot(snap: PriceSnapshot, config: AppConfig): ValidationResult {
   151→  if (snap.region !== config.targetRegion) {
   152→    return { ok: false, error: 'Region mismatch: snapshot.region != config.targetRegion' };
   153→  }
   154→  // ...
   155→}
   156→```

   160→## 1. Главная цель: универсальный движок с автоопределением
   161→
   162→### 1.1. Принцип
   163→
   164→Движок не требует ручного написания spec-схемы под каждый сайт. При добавлении
   165→нового конкурента (URL главной) движок **сам**:
   166→
   167→1. **Probe** — анализирует главную страницу, определяет технологии
   168→2. **Detect tier** — классифицирует сайт (T1-T10) по поведению
   169→3. **Discover price URLs** — находит страницы с ценами (sitemap, BFS, search)
   170→4. **Infer schema** — выводит структуру данных (CSS / data-attrs / JSON / Schema.org)
   171→5. **Generate spec** — создаёт YAML-схему автоматически, помечает как `auto_generated`
   172→6. **Validate** — прогоняет тестовый сбор, проверяет полноту
   173→7. **Self-heal** — при падении парсера пытается перестроить селекторы
   174→
   175→Человек может вмешаться на любом этапе: принять spec, подправить селекторы,
   176→запретить某些 URL. Но **по умолчанию движок автономен**.
   177→
   178→### 1.2. Почему это важно
   179→
   180→- **Масштабируемость:** добавление 10-го конкурента = 1 клик в UI, а не 2 дня работы инженера
   181→- **Устойчивость:** при изменении вёрстки движок сам перестраивает селекторы (с AI-помощью)
   182→- **Качество:** единая унифицированная модель данных, независимо от источника
   183→- **Cost:** 80% сайтов попадают в T1-T2 (auto-detectable без AI), VLM нужен только для edge cases
   184→
   185→### 1.3. Эталонные кейсы для валидации универсальности
   186→
   187→Разведанные сайты используются как **тестовые фикстуры** для движка. Если движок
   188→не может автоматически определить тир и сгенерировать spec для каждого из них —
   189→архитектура недостаточно универсальна.
   190→
   191→| Сайт | Ожидаемый auto-detect | Что проверяем |
   192→|---|---|---|
   193→| veramed-clinic.ru | T1, `tariff_select`, 1 URL = весь прайс | Много блоков с `hidden` атрибутом |
   194→| gemotest.ru | T1, `url_prefix`, 66 сабкатегорий | `data-eec-*` атрибуты |
   195→| helix.ru | T2, `ip_default`, embedded JSON `G.json./api/` | JSON-блобы в SSR HTML |
   196→| altamedplus.ru | T1+discovery, 2 формата таблиц, BFS-crawl | Распределённый прайс по ~294 страницам |
   197→| medsi.ru | Hybrid T1+T3, SEO-блок + SPA labmarket | Multi-strategy per competitor |
   198→| cmd-online.ru | T1+Schema.org, `url_path_segment`, 1510 URL | Schema.org микроразметка |
   199→

   201→
   202→## 2. Probe Engine — автоопределение характеристик сайта
   203→
   204→При добавлении нового сайта (через UI или API) движок запускает **probe
   205→sequence** — серию диагностических запросов для определения характеристик.
   206→
   207→### 2.1. Probe pipeline
   208→
   209→```
   210→new competitor URL
   211→       │
   212→       ▼
   213→┌──────────────────────────────────────────────────────────────┐
   214→│ 1. Fetch robots.txt                                          │
   215→│    → sitemap URLs, Disallow rules, crawl-delay              │
   216→└──────────────────────────────────────────────────────────────┘
   217→       │
   218→       ▼
   219→┌──────────────────────────────────────────────────────────────┐
   220→│ 2. Fetch homepage (curl + page_reader, compare)              │
   221→│    → framework (Bitrix/WordPress/Angular/Vue/React/Next)    │
   222→│    → SSR vs SPA (diff between curl и page_reader)           │
   223→│    → embedded JSON blobs (__NEXT_DATA__, G.json./api/, etc) │
   224→│    → Schema.org microdata presence                          │
   225→│    → currency symbols (₽, руб., р.)                         │
   226→└──────────────────────────────────────────────────────────────┘
   227→       │
   228→       ▼
   229→┌──────────────────────────────────────────────────────────────┐
   230→│ 3. Fetch sitemap(s)                                          │
   231→│    → URL count, categorization by path patterns             │
   232→│    → identify catalog/services/clinics sub-trees            │
   233→└──────────────────────────────────────────────────────────────┘
   234→       │
   235→       ▼
   236→┌──────────────────────────────────────────────────────────────┐
   237→│ 4. Probe catalog page (sample 3-5 URLs from sitemap)        │
   238→│    → price extraction attempts (5 strategies, see 2.3)      │
   239→│    → success rate per strategy                              │
   240→│    → data richness (price/code/term/category availability)  │
   241→└──────────────────────────────────────────────────────────────┘
   242→       │
   243→       ▼
   244→┌──────────────────────────────────────────────────────────────┐
   245→│ 5. Detect region_strategy                                   │
   246→│    → URL patterns with city slugs                           │
   247→│    → cookie/header tests with different regions            │
   248→│    → subdomain checks                                       │
   249→└──────────────────────────────────────────────────────────────┘
   250→       │
   251→       ▼
   252→┌──────────────────────────────────────────────────────────────┐
   253→│ 6. Tier classification (see 2.4)                            │
   254→└──────────────────────────────────────────────────────────────┘
   255→       │
   256→       ▼
   257→┌──────────────────────────────────────────────────────────────┐
   258→│ 7. Generate spec.yaml (auto, marked auto_generated: true)   │
   259→└──────────────────────────────────────────────────────────────┘
   260→       │
   261→       ▼
   262→┌──────────────────────────────────────────────────────────────┐
   263→│ 8. Test scrape (10 URLs) + validation                       │
   264→│    → if pass: spec becomes active                           │
   265→│    → if fail: queue for human review or VLM-assist          │
   266→└──────────────────────────────────────────────────────────────┘
   267→```
   268→
   269→### 2.2. Framework detection heuristics
   270→
   271→```typescript
   272→interface ProbeResult {
   273→  framework: 'bitrix' | 'wordpress' | 'drupal' | 'next' | 'nuxt' | 'angular'
   274→           | 'vue' | 'react-spa' | 'tilda' | 'custom' | 'unknown';
   275→  isSSR: boolean;              // server-side rendering
   276→  hasEmbeddedState: boolean;   // __NEXT_DATA__, window.__INITIAL_STATE__, G.json./api/
   277→  hasSchemaOrg: boolean;       // itemprop="price"
   278→  robotsTxt: ParsedRobots;
   279→  sitemapUrls: string[];
   280→  currencyFormat: '₽' | 'руб.' | 'р.' | 'rub' | 'mixed';
   281→  detectedLanguages: string[];
   282→  antiBotHints: {
   283→    cloudflare?: boolean;
   284→    recaptcha?: boolean;
   285→    jsChallenge?: boolean;
   286→    rateLimitHeaders?: string[];
   287→  };
   288→}
   289→```
   290→
   291→Detection patterns:
   292→
   293→| Признак | Вывод |
   294→|---|---|
   295→| `class="bx-core"` / `/bitrix/` в asset paths | Bitrix |
   296→| `__NEXT_DATA__` script | Next.js (React SSR) |
   297→| `__NUXT__` / `window.__NUXT__` | Nuxt.js (Vue SSR) |
   298→| `ng-version` / `_nghost-` / `_ngcontent-` | Angular |
   299→| `data-v-` attrs (hash) | Vue (no SSR) |
   300→| `data-reactroot` | React (no SSR) |
   301→| `tilda` в meta generator | Tilda |
   302→| `wp-content/` / `wp-includes/` | WordPress |
   303→| `cdn-cgi/challenge-platform` | Cloudflare challenge (T6) |
   304→| `g-recaptcha` / `data-sitekey` | reCAPTCHA (T7) |
   305→
   306→### 2.3. Price extraction strategies (5 подходов)
   307→
   308→Для каждой страницы пробуем **по очереди**, пока не получим ≥1 валидных цен:
   309→
   310→```typescript
   311→const PRICE_STRATEGIES: PriceStrategy[] = [
   312→  // #1 — Schema.org microdata (эталон)
   313→  {
   314→    name: 'schema_org',
   315→    priority: 1,
   316→    regex: /<span[^>]*itemprop="price"[^>]*>([^<]+)<\/span>/,
   317→    currencyRegex: /<meta[^>]*itemprop="priceCurrency"[^>]*content="([^"]+)"/,
   318→    successOn: 'cmd-online.ru'
   319→  },
   320→  // #2 — data-eec-* / data-price / data-product-price attributes
   321→  {
   322→    name: 'data_attributes',
   323→    priority: 2,
   324→    regex: /data-(?:eec-)?price="([^"]+)"/,
   325→    successOn: 'gemotest.ru'
   326→  },
   327→  // #3 — embedded JSON state (Next/Nuxt/Angular SSR)
   328→  {
   329→    name: 'embedded_json',
   330→    priority: 3,
   331→    regex: /"G\.json\.\/api\/[^"]+"\s*:\s*\{"body":(\{)|__NEXT_DATA__[^>]*>(\{)/,
   332→    parser: 'json_path',
   333→    successOn: 'helix.ru'
   334→  },
   335→  // #4 — CSS class patterns (cheerio)
   336→  {
   337→    name: 'css_class',
   338→    priority: 4,
   339→    // auto-discovered via class name heuristics: any class containing 'price'
   340→    selectorCandidates: ['.price', '.product-price', '.service-price', '.analyze-item__price', '*[class*="price"]'],
   341→    successOn: 'altamedplus.ru, veramed-clinic.ru'
   342→  },
   343→  // #5 — SEO text block (минимальные цены)
   344→  {
   345→    name: 'seo_text_block',
   346→    priority: 5,
   347→    regex: /(?:Прием специалиста|Услуги|Описание услуг)[:\s]*([\s\S]+?)<\/p>/,
   348→    itemPattern: /([^.]+?)\s*-\s*(?:от\s+)?(\d[\d\s]*)\s*руб\./,
   349→    successOn: 'medsi.ru'
   350→  },
   351→  // #6 — VLM last resort (screenshot → vision model)
   352→  {
   353→    name: 'vlm_screenshot',
   354→    priority: 99,
   355→    cost: 'high',
   356→    description: 'Делаем screenshot страницы, VLM извлекает JSON с ценами',
   357→    successOn: 'future T8 sites'
   358→  }
   359→];
   360→```
   361→
   362→**Stop condition:** первая стратегия, давшая ≥N валидных цен (N=5 по умолчанию),
   363→становится основной. Остальные — fallback'и в spec.yaml.
   364→
   365→### 2.4. Tier classification matrix
   366→
   367→После probe движок определяет тир по матрице признаков:
   368→
   369→| Признак | → Тир |
   370→|---|---|
   371→| `hasSchemaOrg && priceFound` | **T1+Schema.org** (highest quality, curl only) |
   372→| `framework=='bitrix' && pricesInHTML` | **T1 static** (curl + cheerio) |
   373→| `hasEmbeddedState && pricesInJSON` | **T2 SSR+state** (curl + JSON extract) |
   374→| `isSSR==false && pricesInXHR` | **T3 SPA+API** (page_reader или Playwright intercept) |
   375→| `isSSR==false && !pricesInXHR` | **T4 SPA без API** (Playwright full render) |
   376→| `lazyLoadDetected && !paginationURLs` | **T5 lazy/infinite** (Playwright + scroll) |
   377→| `antiBot.cloudflare` | **T6 light antibot** (stealth + residential proxy) |
   378→| `antiBot.recaptcha \|\| antiBot.datadome` | **T7 heavy antibot** (mobile proxy + VLM) |

   380→| `pdfPriceListDetected` | **T9 PDF** (pdf-parse + LLM normalization) |

   387→function detectRegionStrategy(probe: ProbeResult, sitemapUrls: string[]): RegionStrategy {

   412→  // #5 — IP-based (если cookie не меняют цены, но цены в HTML зависят от гео)

   463→### 3.2. Sitemap discovery (приоритет #1)
   464→
   465→```typescript
   466→async function discoverViaSitemap(baseUrl: string): Promise<DiscoveredUrl[]> {
   467→  // 1. robots.txt → sitemap URLs
   468→  const robots = await fetchRobotsTxt(baseUrl);
   469→  let sitemapUrls = robots.sitemaps;
   470→  
   471→  // 2. Если в robots нет — пробуем стандартные пути
   472→  if (sitemapUrls.length === 0) {
   473→    sitemapUrls = [
   474→      `${baseUrl}/sitemap.xml`,
   475→      `${baseUrl}/sitemap_index.xml`,
   476→      `${baseUrl}/sitemaps/sitemap.xml`
   477→    ].filter(url => await urlExists(url));
   478→  }
   479→  
   480→  // 3. Рекурсивно парсим sitemap index'ы
   481→  const allUrls: string[] = [];
   482→  for (const smUrl of sitemapUrls) {
   483→    const urls = await parseSitemap(smUrl);  // recursively
   484→    allUrls.push(...urls);
   485→  }
   486→  
   487→  // 4. Категоризация URL по path patterns
   488→  return categorizeUrls(allUrls);
   489→}
   490→
   491→function categorizeUrls(urls: string[]): DiscoveredUrl[] {
   492→  return urls.map(url => ({
   493→    url,
   494→    category: classifyUrl(url),  // 'catalog' | 'service' | 'clinic' | 'doctor' | 'article' | 'other'
   495→    hasCitySegment: detectCitySegment(url),
   496→    depth: url.split('/').length - 3,
   497→  }));
   498→}
   499→
   500→function classifyUrl(url: string): UrlCategory {
   501→  const path = new URL(url).pathname;
   502→  if (/\/(services|uslugi|cure|vectors)\//.test(path)) return 'service';
   503→  if (/\/(catalog|katalog|analizy)\//.test(path)) return 'catalog';
   504→  if (/\/(clinics|filialy|branches)\//.test(path)) return 'clinic';
   505→  if (/\/(doctors|vrachi|specialists)\//.test(path)) return 'doctor';
   506→  if (/\/(articles|blog|news|press)\//.test(path)) return 'article';
   507→  return 'other';
   508→}
   509→```
   510→
   511→### 3.3. BFS crawl (fallback, для сайтов без sitemap)
   512→
   513→```typescript
   514→async function discoverViaBFCrawl(seedUrls: string[], options: CrawlOptions): Promise<DiscoveredUrl[]> {
   515→  const visited = new Set<string>();
   516→  const queue: Array<{ url: string; depth: number }> = seedUrls.map(url => ({ url, depth: 0 }));
   517→  const discovered: DiscoveredUrl[] = [];
   518→  
   519→  while (queue.length > 0 && discovered.length < options.maxUrls) {
   520→    const { url, depth } = queue.shift()!;
   521→    if (visited.has(url)) continue;
   522→    visited.add(url);
   523→    
   524→    if (depth > options.maxDepth) continue;
   525→    
   526→    const html = await fetch(url);
   527→    const links = extractLinks(html, url);  // только internal, same-domain
   528→    
   529→    for (const link of links) {
   530→      if (matchesPattern(link, options.linkPattern)) {
   531→        discovered.push({ url: link, category: classifyUrl(link), depth: depth + 1 });
   532→        queue.push({ url: link, depth: depth + 1 });
   533→      }
   534→    }
   535→    
   536→    await delay(options.rateLimitMs);  // politeness
   537→  }
   538→  
   539→  return dedupe(discovered);
   540→}
   541→```

   543→### 3.4. Price URL filtering
   544→
   545→После discovery у нас может быть 10 000+ URL. Нужно отфильтровать только те,
   546→где реально есть цены:
   547→
   548→```typescript
   549→async function filterPriceUrls(urls: DiscoveredUrl[], probe: ProbeResult): Promise<PriceUrl[]> {
   550→  // #1 — heuristic filter по path pattern
   551→  const candidates = urls.filter(u => 
   552→    u.category === 'catalog' || u.category === 'service' || 
   553→    /price|pricelist|cost/i.test(u.url)
   554→  );
   555→  
   556→  // #2 — sample N URLs, проверяем наличие цены через probe
   557→  const sample = sampleN(candidates, 10);
   558→  const probeResults = await Promise.all(
   559→    sample.map(async url => ({
   560→      url,
   561→      hasPrice: await probeHasPrice(url, probe.primaryStrategy)
   562→    }))
   563→  );
   564→  
   565→  const successRate = probeResults.filter(r => r.hasPrice).length / sample.length;
   566→  
   567→  // #3 — если >70% sample содержит цены — берём все кандидаты
   568→  //      если 30-70% — нужна донастройка (human review)
   569→  //      если <30% — strategy не подходит, пробуем следующую
   570→  if (successRate > 0.7) {
   571→    return candidates.map(url => ({ ...url, expected: true }));
   572→  } else if (successRate > 0.3) {
   573→    return candidates.map(url => ({ ...url, expected: 'uncertain' }));
   574→  } else {
   575→    return [];  // trigger next strategy
   576→  }
   577→}
   578→```

   582→## 4. Schema inference — автоматическое понимание структуры
   583→
   584→Даже если мы знаем, что цены есть, нужно понять **структуру** (где name, где
   585→price, где category, где code). Движок использует 3 подхода:
   586→
   587→### 4.1. Heuristic-based (быстрый, без AI)
   588→
   589→```typescript
   590→function inferSchemaHeuristic(html: string, strategy: PriceStrategy): InferredSchema {
   591→  // Для Schema.org — структура задана стандартом
   592→  if (strategy.name === 'schema_org') {
   593→    return {
   594→      price: 'itemprop="price"',
   595→      currency: 'itemprop="priceCurrency"',
   596→      name: 'itemprop="name"',
   597→      availability: 'itemprop="availability"',
   598→      // стандартизировано, не нужно угадывать
   599→    };
   600→  }
   601→  
   602→  // Для data-attrs — анализируем все data-* рядом с ценой
   603→  if (strategy.name === 'data_attributes') {
   604→    const priceElem = findElementWithAttr(html, /data-(?:eec-)?price=/);
   605→    const siblings = getSiblings(priceElem);
   606→    return {
   607→      price: 'data-eec-price',
   608→      name: findAttrInSiblings(siblings, /data-(?:eec-)?name/),
   609→      code: findAttrInSiblings(siblings, /data-(?:eec-)?id/),
   610→      category: findAttrInSiblings(siblings, /data-(?:eec-)?sec/),
   611→    };
   612→  }
   613→  
   614→  // Для CSS-классов — ищем классы по ключевым словам
   615→  if (strategy.name === 'css_class') {
   616→    return {
   617→      price: findClassContaining(html, ['price', 'cost']),
   618→      name: findClassContaining(html, ['name', 'title'], nearElement: priceElem),
   619→      code: findClassContaining(html, ['code', 'art', 'sku']),
   620→      category: findClassContaining(html, ['category', 'section', 'rubric']),
   621→    };
   622→  }
   623→  
   624→  // ...
   625→}
   626→```
   627→
   628→### 4.2. LLM-assisted (для сложных случаев)
   629→
   630→Если эвристики не хватает (например, нет явных классов с "price"), движок
   631→отправляет **фрагмент HTML** в LLM с промптом:
   632→
   633→```
   634→Дан HTML фрагмент карточки товара с медицинского сайта.
   635→Извлеки CSS-селекторы для полей:
   636→- name (название услуги)
   637→- price (цена, число)
   638→- currency (валюта)
   639→- category (категория, если есть)
   640→- code (артикул/код, если есть)
   641→
   642→Верни JSON: { "name": "...", "price": "...", ... }
   643→Только CSS-селекторы, без объяснений.
   644→
   645→HTML:
   646→<div class="med-service-block__type-item">
   647→  <div class="med-service-block__type-item-name">Прием терапевта</div>
   648→  <div class="med-service-block__type-item-price-num">
   649→    <span>1 300 <span class="rub">руб.</span></span>
   650→  </div>
   651→</div>
   652→```
   653→
   654→LLM возвращает:
   655→```json
   656→{
   657→  "name": "div.med-service-block__type-item-name",
   658→  "price": "div.med-service-block__type-item-price-num span:first-child",
   659→  "currency": "span.rub"
   660→}
   661→```
   662→
   663→Результат кэшируется (по хешу HTML-фрагмента) — повторные запросы к тому же
   664→сайту не требуют LLM.
   665→
   666→### 4.3. VLM-assisted (последний рубеж)
   667→
   668→Если LLM не смогла разобрать HTML (например, цены отрисованы canvas'ом или
   669→защищены обфускацией):
   670→
   671→```
   672→1. Playwright делает screenshot страницы
   673→2. VLM получает screenshot + промпт:
   674→   "Извлеки JSON-массив услуг с этой страницы. 
   675→    Каждая услуга: {name, price, category}. 
   676→    Верни только JSON."
   677→3. Ответ валидируется Zod-схемой
   678→4. При успехе — auto-создаётся тикет "VLM fallback сработал, нужна human review"
   679→```
   680→
   681→### 4.4. Schema confidence score
   682→
   683→Каждая inferred schema получает **confidence score** (0-100):
   684→
   685→| Фактор | Баллы |
   686→|---|---|
   687→| Schema.org microdata | +40 |
   688→| data-* атрибуты с явным `price` в имени | +30 |
   689→| Стабильный ID (code, hxid, cat-id) | +20 |
   690→| Цена извлечена на 95%+ sample страниц | +20 |
   691→| Найдена категория/секция | +10 |
   692→| LLM-выведенная схема | -10 (требует валидации) |
   693→| VLM-выведенная схема | -30 (требует валидации) |
   694→
   695→- **≥70** → spec автоматически активируется
   696→- **40-69** → spec помечается `needs_review`, прогоняется тестовый сбор, человек подтверждает
   697→- **<40** → сайт не добавляется, алерт "требуется ручная разборка"

   701→## 5. Adaptive parser — самовосстановление
   702→
   703→Сайты меняются. Движок должен **замечать** поломку парсера и пытаться
   704→самостоятельно перестроить селекторы.
   705→
   706→### 5.1. Health check на каждый scrape
   707→
   708→```typescript
   709→interface ScrapeHealth {
   710→  itemsExtracted: number;
   711→  expectedMin: number;       // из spec
   712→  nullFieldsRate: number;    // % полей = null
   713→  htmlStructureHash: string; // sha256(canonicalize(html))
   714→  structureDiff: number;     // % diff с прошлым успешным сбором
   715→}
   716→
   717→function evaluateHealth(health: ScrapeHealth, spec: Spec): HealthAction {
   718→  // Happy path
   719→  if (health.itemsExtracted >= health.expectedMin && 
   720→      health.nullFieldsRate < 0.1 &&
   721→      health.structureDiff < 0.1) {
   722→    return { action: 'proceed' };
   723→  }
   724→  
   725→  // Вёрстка сильно изменилась
   726→  if (health.structureDiff > 0.3) {
   727→    return { action: 'trigger_reprobe', reason: 'html_structure_changed' };
   728→  }
   729→  
   730→  // Мало items — возможно, селекторы сломались
   731→  if (health.itemsExtracted < health.expectedMin * 0.5) {
   732→    return { action: 'trigger_reprobe', reason: 'items_count_dropped' };
   733→  }
   734→  
   735→  // Много null-полей — структура изменилась частично
   736→  if (health.nullFieldsRate > 0.3) {
   737→    return { action: 'trigger_partial_reprobe', reason: 'fields_missing' };
   738→  }
   739→  
   740→  return { action: 'proceed_with_warning' };
   741→}
   742→```
   743→
   744→### 5.2. Auto-reprobe flow
   745→
   746→При триггере `trigger_reprobe` движок:
   747→
   748→1. **Не удаляет** старую spec, помечает её `deprecated: true`
   749→2. Запускает **полный probe pipeline** (раздел 2) заново
   750→3. Сравнивает новую spec с old:
   751→   - Если strategy та же, изменились только селекторы → auto-update spec
   752→   - Если strategy изменилась (например, сайт переехал с T1 на T3) → human review
   753→4. Прогоняет test scrape, сравнивает с историческими данными:
   754→   - Если цены в ±20% от прошлых — ок, новая spec валидна
   755→   - Если разброс >20% — алерт, возможно новая spec парсит другой тип цен (min vs конкретная)
   756→5. Уведомление в Telegram: «spec для {competitor} auto-updated, причина: {reason}»
   757→
   758→### 5.3. Selector A/B testing
   759→
   760→При сомнениях (confidence 40-69) движок может **параллельно** прогнать 2 версии
   761→селекторов и сравнить результаты:
   762→
   763→```typescript
   764→async function abTestSelectors(url: string, specA: Spec, specB: Spec): Promise<Spec> {
   765→  const [resultA, resultB] = await Promise.all([
   766→    parseWithSpec(url, specA),
   767→    parseWithSpec(url, specB)
   768→  ]);
   769→  
   770→  // Тот, который дал больше валидных items с меньшим nullFieldsRate — победитель
   771→  const scoreA = resultA.items.length * (1 - resultA.nullFieldsRate);
   772→  const scoreB = resultB.items.length * (1 - resultB.nullFieldsRate);
   773→  
   774→  return scoreA >= scoreB ? specA : specB;
   775→}
   776→```

   827→### 6.1. Normalization layer
   828→
   829→Между парсером и БД — слой нормализации:
   830→
   831→```typescript
   832→class Normalizer {
   833→  normalize(item: UniversalPriceItem): NormalizedItem {
   834→    return {
   835→      ...item,
   836→      name: this.normalizeName(item.nameRaw),         // trim, lowercase, collapse spaces
   837→      price: this.normalizePrice(item.priceRaw),      // "1 300 ₽" → 1300
   838→      currency: this.normalizeCurrency(item.currency), // RUB
   839→      category: this.normalizeCategory(item.category), // canonical
   840→      // ...
   841→    };
   842→  }
   843→  
   844→  // LLM-нормализация названий (кэшированная)
   845→  async normalizeNameWithLLM(name: string): Promise<string> {
   846→    const cached = await this.cache.get(`name:${hash(name)}`);
   847→    if (cached) return cached;
   848→    
   849→    const canonical = await this.llm.invoke({
   850→      prompt: `Приведи название медицинской услуги к каноническому виду. 
   851→               Удали лишние слова ("прием", "осмотр", "консультация" оставить).
   852→               Верни только каноническое название.
   853→               Вход: "${name}"`
   854→    });
   855→    
   856→    await this.cache.set(`name:${hash(name)}`, canonical, TTL_30_DAYS);
   857→    return canonical;
   858→  }
   859→}
   860→```
   861→
   862→### 6.2. Cross-source deduplication
   863→
   864→Одна и та же услуга на разных сайтах называется по-разному. Движок использует
   865→**LLM-embedding'и** для сопоставления:
   866→
   867→```typescript
   868→async function findCrossSourceMatches(name: string, allServices: Service[]): Promise<Match[]> {
   869→  const nameEmbedding = await embed(name);  // z-ai embeddings API
   870→  
   871→  const matches = allServices
   872→    .map(s => ({
   873→      service: s,
   874→      similarity: cosineSimilarity(nameEmbedding, s.embedding)
   875→    }))
   876→    .filter(m => m.similarity > 0.85)
   877→    .sort((a, b) => b.similarity - a.similarity);
   878→  
   879→  return matches;
   880→}
   881→```
   882→
   883→Это позволяет строить **сравнительные таблицы** между конкурентами: "Общий анализ
   884→крови: Gemotest 890₽, Helix 835₽, CMD 820₽, Medsi 1300₽ min".

   888→## 7. Pipeline: probe → detect → discover → parse → validate → store
   889→
   890→Полный конвейер для нового конкурента:
   891→
   892→```
   893→[1] Add competitor (URL) ─────────────────────────────────────────┐
   894→                                                                  │
   895→[2] Probe Engine ─────────────────────────────────────────────────┤
   896→    │  • fetch robots.txt                                         │
   897→    │  • fetch homepage (curl + page_reader)                      │
   898→    │  • detect framework, SSR, Schema.org, currency              │
   899→    │  • fetch + parse sitemap                                    │
   900→    │  • sample 5 catalog URLs, test price strategies             │
   901→    │  • detect region_strategy                                   │
   902→    │  • classify tier (T1-T10)                                   │
   903→    ├─────────────────────────────────────────────────────────────┤
   904→    │  Output: ProbeResult + TierClassification                   │
   905→    │                                                              │
   906→[3] Discovery Engine ─────────────────────────────────────────────┤
   907→    │  • sitemap → all URLs                                       │
   908→    │  • categorize URLs (catalog/service/clinic/other)           │
   909→    │  • filter price URLs (sample + probe)                       │
   910→    │  • apply region_strategy filter (scope=moscow)              │
   911→    ├─────────────────────────────────────────────────────────────┤
   912→    │  Output: PriceUrl[] (filtered by region)                    │
   913→    │                                                              │
   914→[4] Schema Inference ─────────────────────────────────────────────┤
   915→    │  • heuristic schema (для schema_org, data_attrs)            │
   916→    │  • LLM-assisted (если эвристики недостаточно)               │
   917→    │  • VLM-assisted (последний рубеж)                           │
   918→    │  • compute confidence score                                 │
   919→    ├─────────────────────────────────────────────────────────────┤
   920→    │  Output: InferredSchema + confidence                        │
   921→    │                                                              │
   922→[5] Spec Generation ──────────────────────────────────────────────┤
   923→    │  • generate spec.yaml (auto_generated: true)                │
   924→    │  • if confidence >= 70 → active                             │
   925→    │  • if 40-69 → needs_review (human approval)                 │
   926→    │  • if < 40 → rejected (manual investigation)                │
   927→    ├─────────────────────────────────────────────────────────────┤
   928→    │  Output: Spec (active | needs_review | rejected)            │
   929→    │                                                              │
   930→[6] Test Scrape ──────────────────────────────────────────────────┤
   931→    │  • fetch 10 sample URLs                                     │
   932→    │  • parse with new spec                                      │
   933→    │  • validate (Zod + business rules)                          │
   934→    │  • compute success rate                                     │
   935→    ├─────────────────────────────────────────────────────────────┤
   936→    │  Output: TestScrapeReport                                   │
   937→    │                                                              │
   938→[7] First Full Scrape ────────────────────────────────────────────┤
   939→    │  • enqueue all filtered URLs to BullMQ                      │
   940→    │  • parallel workers (concurrency from spec)                 │
   941→    │  • per-URL: fetch → parse → normalize → validate            │
   942→    │  • store to Postgres + raw to MinIO                         │
   943→    ├─────────────────────────────────────────────────────────────┤
   944→    │  Output: First batch of PriceSnapshots                      │
   945→    │                                                              │
   946→[8] Ongoing Scrapes (per cron schedule) ──────────────────────────┤
   947→    │  • health check per scrape                                  │
   948→    │  • if health fails → trigger_reprobe (раздел 5)             │
   949→    │  • diff with previous → create PriceSnapshot only on change │
   950→    │  • alerts on price changes >threshold                       │
   951→    └─────────────────────────────────────────────────────────────┘
   952→```

  1072→## 12. Politeness & антиблокировка
  1073→
  1074→1. `robots.txt` уважается — `robots-parser` перед каждым fetch
  1075→2. Rate limit per domain: 1 req / 2 sec по умолчанию, spec переопределяет
  1076→3. `Retry-After` header — если сервер просит ждать, ждём
  1077→4. Exponential backoff: 1s → 2s → 4s → 8s → 16s, max 3 retries
  1078→5. User-Agent: реальный браузерный, ротация между 5-10 актуальными
  1079→6. Conditional requests: `If-Modified-Since`, `If-None-Match`
  1080→7. HTTP/2 по умолчанию
  1081→8. Jitter: random ±20% к задержке
  1082→9. Circuit breaker: 5 ошибок подряд → пауза 10 мин на домен

  1086→## 13. Resilient parsing — multi-selector fallback
  1087→
  1088→В auto-generated spec.yaml:
  1089→
  1090→```yaml
  1091→parsers:
  1092→  - name: primary
  1093→    selector: 'span[itemprop="price"]'
  1094→    confidence: 90
  1095→  - name: fallback_1
  1096→    selector: 'div.analyze-item__price'
  1097→    confidence: 70
  1098→    trigger: primary_items < expected * 0.5
  1099→  - name: fallback_2
  1100→    selector: 'div[class*="price"]'
  1101→    confidence: 50
  1102→    trigger: fallback_1_items < expected * 0.5
  1103→  - name: vlm_last_resort
  1104→    type: vlm_screenshot
  1105→    confidence: 30
  1106→    trigger: all_previous_failed
  1107→```
  1108→
  1109→При каждом парсинге пробуем primary, если fails — fallback_1, и т.д. Каждый
  1110→fallback логируется с причиной. 3 отказа подряд → incident.

  1355→## 18. Anti-patterns (чего НЕ делать)
  1356→
  1357→1. ❌ Hardcoded селекторы в коде — только в spec-файлах
  1358→2. ❌ Spec per competitor без auto-detection — движок должен быть universal
  1359→3. ❌ `Float` для денег — только `Decimal`
  1360→4. ❌ Sync sleep в hot path — только Promise-based
  1361→5. ❌ Один try-catch на весь pipeline — per-stage
  1362→6. ❌ Логи без `traceId`
  1363→7. ❌ Puppeteer вместо Playwright
  1364→8. ❌ Без raw-lake — теряем возможность пере-парсинга
  1365→9. ❌ Retry без exponential backoff → гарантированный ban
  1366→10. ❌ Без robots.txt проверки
  1367→11. ❌ Selenium — устарел
  1368→12. ❌ BeautifulSoup/Scrapy (Python) — не тащить второй язык
  1369→13. ❌ Прямые запросы к DB из парсера — только через репозиторий
  1370→14. ❌ Magic numbers — все лимиты/timeout'ы в конфиге
  1371→15. ❌ VLM как primary стратегия — только fallback
  1372→16. ❌ Без confidence score — не знаем, можно ли доверять spec'у
  1373→17. ❌ Auto-generated spec без human override — всегда должна быть кнопка «edit spec»
